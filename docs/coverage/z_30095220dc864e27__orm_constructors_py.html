753 """Add additional WHERE criteria to the load for all occurrences of 

754 a particular entity. 

755 

756 .. versionadded:: 1.4 

757 

758 The :func:`_orm.with_loader_criteria` option is intended to add 

759 limiting criteria to a particular kind of entity in a query, 

760 **globally**, meaning it will apply to the entity as it appears 

761 in the SELECT query as well as within any subqueries, join 

762 conditions, and relationship loads, including both eager and lazy 

763 loaders, without the need for it to be specified in any particular 

764 part of the query. The rendering logic uses the same system used by 

765 single table inheritance to ensure a certain discriminator is applied 

766 to a table. 

767 

768 E.g., using :term:`2.0-style` queries, we can limit the way the 

769 ``User.addresses`` collection is loaded, regardless of the kind 

770 of loading used:: 

771 

772 from sqlalchemy.orm import with_loader_criteria 

773 

774 stmt = select(User).options( 

775 selectinload(User.addresses), 

776 with_loader_criteria(Address, Address.email_address != 'foo')) 

777 ) 

778 

779 Above, the "selectinload" for ``User.addresses`` will apply the 

780 given filtering criteria to the WHERE clause. 

781 

782 Another example, where the filtering will be applied to the 

783 ON clause of the join, in this example using :term:`1.x style` 

784 queries:: 

785 

786 q = session.query(User).outerjoin(User.addresses).options( 

787 with_loader_criteria(Address, Address.email_address != 'foo')) 

788 ) 

789 

790 The primary purpose of :func:`_orm.with_loader_criteria` is to use 

791 it in the :meth:`_orm.SessionEvents.do_orm_execute` event handler 

792 to ensure that all occurrences of a particular entity are filtered 

793 in a certain way, such as filtering for access control roles. It 

794 also can be used to apply criteria to relationship loads. In the 

795 example below, we can apply a certain set of rules to all queries 

796 emitted by a particular :class:`_orm.Session`:: 

797 

798 session = Session(bind=engine) 

799 

800 @event.listens_for("do_orm_execute", session) 

801 def _add_filtering_criteria(execute_state): 

802 

803 if ( 

804 execute_state.is_select 

805 and not execute_state.is_column_load 

806 and not execute_state.is_relationship_load 

807 ): 

808 execute_state.statement = execute_state.statement.options( 

809 with_loader_criteria( 

810 SecurityRole, 

811 lambda cls: cls.role.in_(['some_role']), 

812 include_aliases=True 

813 ) 

814 )

817 event will intercept all queries emitted using the

819 and are not attribute or relationship loads a custom 

820 :func:`_orm.with_loader_criteria` option is added to the query. The 

821 :func:`_orm.with_loader_criteria` option will be used in the given 

822 statement and will also be automatically propagated to all relationship 

823 loads that descend from this query. 

824

826 argument. The given class will expand to include all mapped subclass 

827 and need not itself be a mapped class. 

828 

829 .. tip:: 

830 

831 When using :func:`_orm.with_loader_criteria` option in 

832 conjunction with the :func:`_orm.contains_eager` loader option, 

833 it's important to note that :func:`_orm.with_loader_criteria` only 

834 affects the part of the query that determines what SQL is rendered 

835 in terms of the WHERE and FROM clauses. The 

836 :func:`_orm.contains_eager` option does not affect the rendering of 

837 the SELECT statement outside of the columns clause, so does not have 

838 any interaction with the :func:`_orm.with_loader_criteria` option. 

839 However, the way things "work" is that :func:`_orm.contains_eager` 

840 is meant to be used with a query that is already selecting from the 

841 additional entities in some way, where 

842 :func:`_orm.with_loader_criteria` can apply it's additional 

843 criteria.

847 option will affect the way in which the JOIN is rendered:: 

848 

849 stmt = select(A).join(A.bs).options( 

850 contains_eager(A.bs), 

851 with_loader_criteria(B, B.flag == 1) 

852 ) 

853

855 affect the ON clause of the JOIN that is specified by

857 :func:`_orm.contains_eager` option has the effect that columns from 

858 ``B`` are added to the columns clause:: 

859 

860 SELECT 

861 b.id, b.a_id, b.data, b.flag, 

862 a.id AS id_1, 

863 a.data AS data_1 

864 FROM a JOIN b ON a.id = b.a_id AND b.flag = :flag_1 

865 

866

868 statement has no effect on the behavior of the 

869 :func:`_orm.with_loader_criteria` option. If the 

870 :func:`_orm.contains_eager` option were omitted, the SQL would be 

871 the same as regards the FROM and WHERE clauses, where 

872 :func:`_orm.with_loader_criteria` continues to add its criteria to 

873 the ON clause of the JOIN. The addition of 

874 :func:`_orm.contains_eager` only affects the columns clause, in that 

875 additional columns against ``b`` are added which are then consumed 

876 by the ORM to produce ``B`` instances. 

877 

878 .. warning:: The use of a lambda inside of the call to 

879 :func:`_orm.with_loader_criteria` is only invoked **once per unique 

880 class**. Custom functions should not be invoked within this lambda. 

881 See :ref:`engine_lambda_caching` for an overview of the "lambda SQL" 

882 feature, which is for advanced use only. 

883 

884 :param entity_or_base: a mapped class, or a class that is a super 

885 class of a particular set of mapped classes, to which the rule 

886 will apply. 

887 

888 :param where_criteria: a Core SQL expression that applies limiting 

889 criteria. This may also be a "lambda:" or Python function that 

890 accepts a target class as an argument, when the given class is 

891 a base with many different mapped subclasses. 

892 

893 .. note:: To support pickling, use a module-level Python function to 

894 produce the SQL expression instead of a lambda or a fixed SQL 

895 expression, which tend to not be picklable. 

896 

897 :param include_aliases: if True, apply the rule to :func:`_orm.aliased`

899

901 loaders such as lazy loaders. This indicates that the 

902 option object itself including SQL expression is carried along with 

903 each loaded instance. Set to ``False`` to prevent the object from 

904 being assigned to individual instances. 

905 

906 

907 .. seealso:: 

908 

909 :ref:`examples_session_orm_events` - includes examples of using 

910 :func:`_orm.with_loader_criteria`. 

911 

912 :ref:`do_orm_execute_global_criteria` - basic example on how to 

913 combine :func:`_orm.with_loader_criteria` with the 

914 :meth:`_orm.SessionEvents.do_orm_execute` event.

920 it returns the identical SQL every time given a particular class. 

921 

922 .. versionadded:: 1.4.0b2 

923 

924 """

930 propagate_to_loaders,
