495 def statement(self) -> Union[Select[_T], FromStatement[_T]]: 

496 """The full SELECT statement represented by this Query. 

497 

498 The statement by default will not have disambiguating labels 

499 applied to the construct unless with_labels(True) is called 

500 first. 

501 

502 """ 

503 

504 # .statement can return the direct future.Select() construct here, as 

505 # long as we are not using subsequent adaption features that 

506 # are made against raw entities, e.g. from_self(), with_polymorphic(), 

507 # select_entity_from(). If these features are being used, then 

508 # the Select() we return will not have the correct .selected_columns 

509 # collection and will not embed in subsequent queries correctly. 

510 # We could find a way to make this collection "correct", however 

511 # this would not be too different from doing the full compile as 

512 # we are doing in any case, the Select() would still not have the 

513 # proper state for other attributes like whereclause, order_by, 

514 # and these features are all deprecated in any case. 

515 # 

516 # for these reasons, Query is not a Select, it remains an ORM 

517 # object for which __clause_element__() must be called in order for 

518 # it to provide a real expression object. 

519 # 

520 # from there, it starts to look much like Query itself won't be 

521 # passed into the execute process and won't generate its own cache 

522 # key; this will all occur in terms of the ORM-enabled Select. 

523 if not self._compile_options._set_base_alias: 

524 # if we don't have legacy top level aliasing features in use 

525 # then convert to a future select() directly 

526 stmt = self._statement_20(for_statement=True) 

527 else: 

528 stmt = self._compile_state(for_statement=True).statement 

529 

530 if self._params:

2238 :class:`_query.Query`. 

2239 

2240 **Simple Relationship Joins** 

2241 

2242 Consider a mapping between two classes ``User`` and ``Address``, 

2243 with a relationship ``User.addresses`` representing a collection 

2244 of ``Address`` objects associated with each ``User``. The most 

2245 common usage of :meth:`_query.Query.join` 

2246 is to create a JOIN along this 

2247 relationship, using the ``User.addresses`` attribute as an indicator 

2248 for how this should occur:: 

2249 

2250 q = session.query(User).join(User.addresses) 

2251 

2252 Where above, the call to :meth:`_query.Query.join` along 

2253 ``User.addresses`` will result in SQL approximately equivalent to:: 

2254 

2255 SELECT user.id, user.name 

2256 FROM user JOIN address ON user.id = address.user_id 

2257 

2258 In the above example we refer to ``User.addresses`` as passed to 

2259 :meth:`_query.Query.join` as the "on clause", that is, it indicates 

2260 how the "ON" portion of the JOIN should be constructed. 

2261 

2262 To construct a chain of joins, multiple :meth:`_query.Query.join` 

2263 calls may be used. The relationship-bound attribute implies both 

2264 the left and right side of the join at once:: 

2265 

2266 q = session.query(User).\ 

2267 join(User.orders).\ 

2268 join(Order.items).\ 

2269 join(Item.keywords) 

2270 

2271 .. note:: as seen in the above example, **the order in which each 

2272 call to the join() method occurs is important**. Query would not, 

2273 for example, know how to join correctly if we were to specify 

2274 ``User``, then ``Item``, then ``Order``, in our chain of joins; in 

2275 such a case, depending on the arguments passed, it may raise an 

2276 error that it doesn't know how to join, or it may produce invalid 

2277 SQL in which case the database will raise an error. In correct 

2278 practice, the 

2279 :meth:`_query.Query.join` method is invoked in such a way that lines 

2280 up with how we would want the JOIN clauses in SQL to be 

2281 rendered, and each call should represent a clear link from what 

2282 precedes it. 

2283 

2284 **Joins to a Target Entity or Selectable** 

2285 

2286 A second form of :meth:`_query.Query.join` allows any mapped entity or 

2287 core selectable construct as a target. In this usage, 

2288 :meth:`_query.Query.join` will attempt to create a JOIN along the 

2289 natural foreign key relationship between two entities:: 

2290 

2291 q = session.query(User).join(Address) 

2292 

2293 In the above calling form, :meth:`_query.Query.join` is called upon to 

2294 create the "on clause" automatically for us. This calling form will 

2295 ultimately raise an error if either there are no foreign keys between 

2296 the two entities, or if there are multiple foreign key linkages between 

2297 the target entity and the entity or entities already present on the 

2298 left side such that creating a join requires more information. Note 

2299 that when indicating a join to a target without any ON clause, ORM 

2300 configured relationships are not taken into account. 

2301 

2302 **Joins to a Target with an ON Clause** 

2303 

2304 The third calling form allows both the target entity as well 

2305 as the ON clause to be passed explicitly. A example that includes 

2306 a SQL expression as the ON clause is as follows:: 

2307 

2308 q = session.query(User).join(Address, User.id==Address.user_id) 

2309 

2310 The above form may also use a relationship-bound attribute as the 

2311 ON clause as well:: 

2312 

2313 q = session.query(User).join(Address, User.addresses) 

2314 

2315 The above syntax can be useful for the case where we wish 

2316 to join to an alias of a particular target entity. If we wanted 

2317 to join to ``Address`` twice, it could be achieved using two 

2318 aliases set up using the :func:`~sqlalchemy.orm.aliased` function:: 

2319 

2320 a1 = aliased(Address) 

2321 a2 = aliased(Address) 

2322 

2323 q = session.query(User).\ 

2324 join(a1, User.addresses).\ 

2325 join(a2, User.addresses).\ 

2326 filter(a1.email_address=='ed@foo.com').\ 

2327 filter(a2.email_address=='ed@bar.com')

2332 

2333 a1 = aliased(Address) 

2334 a2 = aliased(Address) 

2335 

2336 q = session.query(User).\ 

2337 join(User.addresses.of_type(a1)).\ 

2338 join(User.addresses.of_type(a2)).\ 

2339 filter(a1.email_address == 'ed@foo.com').\ 

2340 filter(a2.email_address == 'ed@bar.com')
